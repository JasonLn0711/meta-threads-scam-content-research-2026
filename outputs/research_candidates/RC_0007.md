Title:
Reviewer-hour value separates guarantee fast lanes from result-display routing lanes

Observation:
Batch 0008 confirmed that `strong_source_priority` produced high-value reviewed
candidates with low review burden, while result-display lanes remained either
uncertain, thread-expensive, or hard-negative calibration.

Hypothesis:
The system should treat guarantee/certainty anchors with executable transition
structure as a fast reviewer-priority lane, while routing result-display patterns
into boundary, slow-context, or hard-negative calibration lanes. This should
increase high-value candidate discovery per reviewer hour without turning the
system into an automated classifier.

Next experiment:
Design a context-gating policy check that routes future structured metadata into
four review lanes before human review:

```text
strong_source_priority -> fast priority review
result_display_low_context_transition -> cheap boundary triage
result_display_thread_required -> slow context review or capped diagnostic sample
result_display_clean_holdout -> hard-negative calibration
```

Do not promote a new sparse feature from Batch 0008. The latest discrepancy
report has no valid `missed_pattern` group with size >= 3.
