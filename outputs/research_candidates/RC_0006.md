# RC_0006: Context Gating For Result-Display Review Burden

## Observation

Batch 0007 split result-display cases into four reviewer-burden lanes.

The split did not increase positive yield:

```text
batch_0007_svs: 0.0
scam_count: 0
uncertain_count: 9
non_scam_count: 3
```

But it did separate review cost:

```text
low_context_transition:
  review_time: 41.333333
  needs_thread_rate: 0.0
  second_review_rate: 0.0

thread_required:
  review_time: 49.0
  needs_thread_rate: 1.0
  second_review_rate: 1.0

emotional_thread_required:
  review_time: 51.0
  needs_thread_rate: 1.0
  second_review_rate: 1.0

clean_holdout:
  review_time: 37.0
  non_scam_count: 3
```

## Hypothesis

`成果展示` should not be treated as a high-yield source signal by itself.

It is better used as a context-gating family:

```text
成果展示 + visible transition + no needs_thread -> single-item review
成果展示 + needs_thread -> thread context review
成果展示 + needs_thread + 情緒操控 -> thread context plus second-review queue
成果展示 only -> hard-negative calibration
```

## Next Experiment

Run a capped Batch 0008 that keeps collection small and tests whether the new
context gates reduce reviewer effort without losing high-value candidates.

Suggested design:

```text
A: strong_source_priority control
B: result_display_low_context_transition
C: result_display_thread_required
D: result_display_clean_holdout
```

Measure:

- candidates reviewed per reviewer hour;
- full-thread-reading rate;
- second-review rate;
- uncertain rate;
- hard-negative protection;
- SVS change by lane.
